s14/a17 (recipe make-the-reactiveagents-chat). The deep-research write/review "thin report" has TWO opposite root causes that present identically, and the trace forced a 3-point calibration:

- **Over-feed** (d164): one UNSCOPED single-section writer folded ALL 11 fetched source bodies inline → 137KB = 107% of num_ctx → Ollama silently truncated the tail (source index + CSS spec) → hollow + visually plain. a14's per-source chunking was SCOPED-only, so the served single-section synthesis path bypassed it.
- **Under-feed** (d167, my own miscalibration): replacing the fold with a compact index + ~2400-char figure LEADS + a load_source nudge STARVED the writer — because the raw-file writer is raw-emission BY DESIGN (d49: E4B can't reliably emit tool-call JSON) so it CANNOT pull on demand. Same thin symptom, opposite cause.
- **RIGHT (d170)**: with a source-count ceiling (max_sources=8) + per-section scoping, PUSH the FULL figure-bearing bodies of each section's FEW (2-4) assigned sources (~20-40KB), per-source = writer budget, TOTAL capped to ~0.6×char-envelope so it never truncates. This is the good-run calibration (writer inputs 20-39KB, dozens of figures). The REVIEWER keeps the compact index + load_source PULL (it CAN call tools).

**Why:** "bounded" is necessary but a bounded feed can still be STARVED. The lever is feed CONTENT sized to the model's pull-ability, not just feed SIZE.

**How to apply:** route the source feed by role — writer gets full bodies of its scoped sources (render_scoped_sources, total-capped), reviewer gets the index (render_scoped_source_index) + pulls. Prove on the trace BOTH (a) each writer input CONTAINS its assigned sources' full body, AND (b) no call ≥ num_ctx-1. Keep any convo-window backstop OBSERVABLE + dormant (the tool/feed must bound by construction, never an app truncation cap = the silent-truncation defect itself).

**s15/a12 (d184 WRITE-FROM-NOTES) — the next layer on this same feed.** Making the durable research NOTES (per-source summary+key_claims, keyed by the same [S#]) the per-section WRITER's PRIMARY substrate is a clean late-pass in `run_section_write_phase` (`_inject_section_notes`, run AFTER `_ensure_source_coverage`+`_flag_unsupported_sections` so each writer's scoped notes match its FINAL source_ids). Two traps that shaped it: (1) the same writer-can't-pull reality (d49) means "pull a full chunk via load_source when a note is thin" is, FOR THE WRITER, realized as a PREP-TIME PUSH — push the notes + KEEP the proven full-source body push as the figure/thin-note grounding; only the REVIEWER does the real on-demand pull. So write-from-notes does NOT remove the goldilocks full-body push (shrinking it lives in runtime.py + risks d167 thinness again — out of an agentic.py-scoped action; flag it as the next lever if a14's served render shows scoped sources still dominating). (2) Detect writer-vs-review nodes by the framework review-id SUFFIX, NOT role/tool — `_normalize_write_dag` stamps `tool=file_write`/`role=None` onto EVERY node (incl. reviews) after `_inject_section_reviews`, so role is not a discriminator post-normalize. Pitfall hit: the ReactiveAgents working tree carried HEAVY uncommitted prior-leg work (d168 etc.), so `git stash`-vs-HEAD MISLEADS when isolating one leg's effect (a "my change broke it" was actually a pre-existing working-tree d168 rewrite of `_flag_unsupported_sections` — passes on HEAD, already red in the tree) — diff the SPECIFIC function, not the whole file, and a function you never touched differing from HEAD = a prior uncommitted leg.

Also load-bearing this action: section COHERENCE can't be judged by counts — run2 passed "figures+table present" while half the sections were empty UNSUPPORTED stubs + duplicate Analysis/Costs. The lead must be a SYNTHESIS (never source-gated); drop sourceless non-lead sections (don't stub them — the dropped empties WERE the duplicates); decompose facets must be mutually exclusive (an analysis-of-X facet overlapping a dedicated-X facet starves the dedicated one); the anchored reviewer's file_update can corrupt a table cell (partial str-replace → run-on `<td>`), so HTML edits must be structure-safe/repaired. ALWAYS read the rendered report + assert coherence (0 empty, 0 dup families, 0 malformed cells, real lead), per [[runtime-proof-needs-manual-evidence-review]].
