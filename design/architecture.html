Decision ledger — active & load-bearing (read-only reference)

  • d1 — Runtime is pure phi4-mini (Ollama http://localhost:11435, OpenAI-compatible + native API) for ALL reasoning/planning. Zero Claude in the agent runtime; the agent is standalone.
  • d2 — Concurrency is in-process: lightweight RxPY/asyncio reactive layer (NOT eda-base3's broker/pool HTTP services). Agents launched & tracked in-process as tasks/coroutines (NOT claude.exe shell-forking).
  • d3 — Memory uses an in-process CPU embedder all-MiniLM-L6-v2 (384d, ONNX/int8) + sqlite-vec; structure-first then hybrid BM25+dense, RRF-fused. Forbidden: Ollama/nomic embeddings (GPU thrash vs phi4-mini).
  • d4 — Token discipline is Claude-Code-style context management (in-window + auto-compaction + manual compact), NOT a hard per-call pass/fail gate. Served by selective recall & the context-scoping principle (d10).
  • d5 — (SUPERSEDED by d9) auto-compile of specializations. do NOT reflect d5
  • d6 — Planner autonomy: the planner builds its own DAG (e.g. research→write-md→write-html) with no hard-coded task prompts; web access is free, no-key (DuckDuckGo HTML search + page fetch/extract).
  • d7 — Build order & acceptance: (1) this interactive HTML design doc, user-agreed FIRST; (2) core phi library; (3) reactive+tool, memory, specialization engine, planner+runtime; (4) full chat UI; (5) end-to-end demo. Adapt-bar: the agent autonomously trains TWO specializations (markdown + html) and writes a detailed report on one topic via each (research→write→write, planner-chosen).
  • d8 — Host constraints (Windows): Python+uv per-component .venv; verify gates are file-based / runtime-capture only (never shell-command gates); never name/image-wide python/node kills (stop only specific PIDs you started); GPU is shared — CPU embeddings + Ollama keep_alive unload to avoid contention.
  • d9 — Specialization lifecycle (corrects d5): the UI defines a specialization → it web-researches the "how" → user-approval compile gate (HITL, not silent auto-compile) → compiled spec stored in a lookup/registry, reused later by sub-agents. No learnings flowback.
  • d10 — Context-scoping = memory optimization. The planner is aware ONLY of the abstract factory + the specialization lookup index (never full spec bodies or heavy phased prompts). Each launched sub-agent is aware ONLY of its task + the single compiled spec it loads. Keeps phi's small window lean while the task still completes autonomously.

Rejected options (banned — must not appear in the design)

  • Reproducing eda-base3's broker/pool HTTP microservices and claude.exe shell-forking spawn model — this is a single-host, single-process standalone app; in-process RxPY + in-process agent tasks are lighter and match "launch agents and track them, not multiple shells."
  • Ollama-hosted (nomic-embed-text) embeddings for memory — phi4-mini and nomic offload each other on the shared GPU (thrash); CPU MiniLM in-process eliminates the contention.
Read-only grounding: eda-base3's skills-architecture is referenced ONLY for the specialization-lookup + role-scoped-context pattern (d9/d10). ReactiveAgents is a STANDALONE, non-coding, real-world-data agent — not a clone of the Claude-Code harness. Do not import its broker/pool/shell model.
